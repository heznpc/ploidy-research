# Privacy Policy — Ploidy MCP Server

**Status: draft.** This document covers a user-operated deployment
of the open-source Ploidy MCP server. If you are using Ploidy as
a hosted service, the operator of that deployment is the data
controller and their own privacy notice governs — this document
describes what the upstream software does, not any specific
operator's practices.

**Effective date: 2026-04-23**

## Summary

Ploidy is a self-hosted tool for running structured debates between
same-model LLM sessions with different context depths. It holds only
what it needs to run those debates and give you back your history.
It is not an advertising product, does not profile users, and does
not share data with third parties beyond the model provider you
point it at.

## Data Ploidy stores

The MCP server persists, locally in a SQLite file (default
`~/.ploidy/ploidy.db`) on the machine you run it on:

| Category | What | Why |
|---|---|---|
| Debate inputs | The prompt you submitted, the optional context documents you attached | So past debates are reviewable from `ploidy-history` and `ploidy-dashboard` |
| Debate outputs | Positions / challenges / synthesis / convergence points generated by the model | Same — history + re-rendering |
| Session metadata | Debate ids, timestamps, status, effort level, model/injection-mode settings | Diagnostics + paper reproducibility |
| OAuth state (if enabled) | Registered client metadata (`client_id`, redirect URIs, client name), short-lived authorization codes, access and refresh tokens (hashed or opaque) | Required to implement OAuth 2.0 |
| Metrics | Aggregate Prometheus counters (no per-user data) | `/metrics` scrape |

## Data Ploidy does NOT store

- **Model-provider credentials** beyond what the process environment
  provides. `PLOIDY_API_KEY` / `ANTHROPIC_API_KEY` are read from env
  vars and never written to disk by Ploidy.
- **User accounts** beyond OAuth `client_id` records. There is no
  email, password, real name, or billing information collected.
- **Client IP addresses or user-agents.** Standard HTTP access logs
  live wherever the operator configures (stdout by default); Ploidy
  does not persist them.

## Third parties

Ploidy sends the prompt and context documents you submit to the
model provider configured by the operator (`PLOIDY_API_BASE_URL`,
typically Anthropic's OpenAI-compatible endpoint). That provider's
data policy applies to the outbound text.

Ploidy does not use analytics services, ad networks, or third-party
trackers.

## Retention

- **Debates** are retained until the operator deletes them via
  `ploidy-history`, the MCP `debate_delete` tool, or the configured
  `PLOIDY_RETENTION_DAYS` automatic purge.
- **OAuth authorization codes** expire in 5 minutes and are purged
  on the retention tick (runs hourly by default).
- **OAuth access tokens** default to 1-hour expiry; refresh tokens
  do not expire but are revoked on use (rotation) or on explicit
  `/revoke` call.
- The operator can wipe all data at any time by stopping the server
  and deleting `PLOIDY_DB_PATH`.

## User rights

Because Ploidy is self-hosted on your own machine:

- **Access**: `ploidy-history show <id>` reads any stored debate.
- **Deletion**: the MCP `debate_delete` tool removes a single
  debate; deleting the SQLite file wipes everything.
- **Portability**: the SQLite file is a standard database readable
  by any SQLite client.
- **Correction**: there is no "correction" step — debate outputs
  are a record of what the model produced, not user-authored
  facts. Delete and re-run is the supported flow.

## Security

- Default bearer-token auth (`PLOIDY_TOKENS`) uses `hmac.compare_digest`
  for constant-time comparison.
- OAuth 2.0 mode (`PLOIDY_AUTH_MODE=oauth`) enforces PKCE S256 and
  single-use authorization codes.
- The database file is created with `0o700` (owner-only) permissions.
- Rate limiting (`PLOIDY_RATE_CAPACITY`) is available per-tenant.

No software is perfectly secure. Security issues should be reported
privately to the maintainer.
